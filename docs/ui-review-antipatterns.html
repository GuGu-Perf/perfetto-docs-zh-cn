Perfetto UI Review 反模式

这是一份 UI 反模式及推荐替代方案的规范目录，源自真实的 code-review 反馈——维护者在对其他贡献者提交的 ui/src/ pull request 中反复给出的意见。

目的： 供 AI agent（或人类）在 code-review 或编写代码时 阅读， 以便我们预先避免跨贡献反复出现的错误和无效假设，而不是在每个 PR 中 反复讨论。

如何使用：

  • 每个条目格式为 ❌ 反模式 → ✅ 修复，附带一行 原因 以及一个或多个 可供查阅完整上下文的历史 PR 编号（#NNNN）。
  • 🔁 标记表示跨多个 PR 反复出现的规则——应给予最大权重； 这些是维护者最强烈、最反复表达的偏好。
  • 本文补充（并深入于）AGENTS-ui.md 中的 "Code Review Pet Peeves" 部分。当两者重叠时，两者都是权威的。

元原则，几乎贯穿以下所有内容：不要过度工程化，重用已有的东西， 状态只存一处，让失败可见。有疑问时，选择更简单、更小、更无聊的变更。

1. Mithril 与渲染

  • 🔁 ❌ 手动调用 m.redraw() / raf.scheduleFullRedraw() → ✅ 依赖自动 重绘。Mithril 在 (1) 它注册的 DOM 事件处理器和 (2) trace-processor 查询完成后会自动安排重绘。手动重绘仅在真正带外异步 （setTimeout、fetch、yield）时需要。提交代码中的绝大多数手动 重绘都是不必要的——删除它们。如果你认为需要手动重绘，首先找出 UI 为何 不更新；这通常指向别处的真正 bug，而非缺少重绘。当确实需要时，在事件 发生点调用 m.redraw()（它不依赖于 app），而非 app.raf.scheduleFullRedraw()。 (#4769, #4464, #4357, #4226, #3999, #3693, #3124, #2615, #1097, #5595, #5436, #5284, #5243)

  • 🔁 ❌ 将 attrs、设置或查询结果缓存到私有成员字段中 → ✅ 每次新鲜 读取。Mithril 将 attrs 传递给 oncreate/onupdate/view；从那里 使用它们。设置存储已具有缓存支持——每次访问都读值，确保用户更改时 不会过时。展示组件必须容忍其数据源每帧都可能变化。 (#3857, #6031, #4737)

  • ❌ requestAnimationFrame / setTimeout 时机 hack，希望 DOM 在"下一帧" 就绪 → ✅ 使用恰当的渲染后回调。用 rAF 规避 ResizeObserver 循环警告 是代码异味——重构为正确的渲染后回调。 (#5595, #5243, #4226, #3780, #3522, #2486)

2. 状态管理

  • 🔁 ❌ 将异步数据加载到组件的本地状态中 → ✅ 使用 AsyncLimiter 或 AsyncContent 模式。加载 trace 处理器查询结果、网络 fetches 或慢速 wasm 调用到本地状态会导致竞态条件（陈旧数据覆盖新鲜数据）。改用 AsyncLimiter（在 oncreate 中排队，在 onremove 中取消，每次 render 渲染陈旧数据时使用 retainOn），或在 TrackEventDetailsPanel.load() 中加载，它自动取消过期的加载。使用 AsyncLimiter.isRunning 而非自定义标志来守卫并发。（注意：isFresh 始终为 true，除非你使用了 retainOn 键。） (#4464, #4582, #4737, #5436, #4192)

  • ❌ 维护需要手动保持同步的派生状态 → ✅ 从单一数据源派生。不要通过 匹配选项来反向推导状态——持久化实际选择（例如 preset 名称）并重新 应用它。为异构项使用统一的 selection 列表，而非平行的 selection 状态。 (#3693, #3780, #3999)

  • 🔁 ❌ 将 UI 时间戳/持续时间存储为 JS number → ✅ 使用 bigint。JS number 是 double，在 trace time 的大动态范围内会丢失精度。将 SQL 时间列 类型化为 LONG/LONG_NULL，使其物化为 bigint（无需手动 BigInt() / 无精度损失），并使用 Time.fromRaw() 而非强制转换。 (#2503, #4767, #4769)

  • ❌ 按人类可读的显示名称查询行 → ✅ 按稳定的有类型列查询（例如 type = 'suspend_resume'，而非 name = "Suspend/Resume Latency"）。 显示字符串是脆弱的。*(#2149)*

  • ❌ 忘记在 onActivate()、onTraceLoad() 或其他生命周期方法中 await 异步函数调用。调用异步方法而不 await 或 .then() 的正当理由极为罕见， 因为这会创建 fire-and-forget 语义，导致竞态。如果确实需要，必须用注释 记录。*(#2353, #1758)*

  • ❌ 为可关闭提示/持久化布局使用临时 localStorage 键 → ✅ 使用设置 系统（并用 Zod 解析未知的持久化对象，使新增字段可选以保证向后兼容）。 让 banner 可永久关闭；将一次性提示与永久可发现的按钮配对，而非重复的 侵入式 banner。*(#5761, #2772, #3660, #3185)*

3. 架构与分层

  • 🔁 ❌ src/core 或 public/ API 接口中的厂商/产品/plugin 特定代码 → ✅ 将特定逻辑推入 plugin；保持核心接口通用。这是错误的分层。同样 适用于 chrome：不要向共享侧边栏添加 plugin 特定部分——通过 subtab/submenu 将导航保留在 plugin 自己的页面内。将 plugin 特定常量（track kind）、 工厂和 schema 保留在所属 plugin 内。 (#5284, #5153, #1624, #1241)

  • ❌ 将纯粹的通用库代码（track 实现、面板）包装为 plugin → ✅ 放入 src/core 或 src/public。Plugin 用于外部或可选功能。通用功能不应 是可选的；如果一个核心 tab 依赖于某项功能，它就不该是一个 plugin。 (#5153, #4991, #4844)

  • ❌ 依赖整体 App / TraceContext 对象 → ✅ 只依赖所需的具体接口 （TraceImpl、SelectionManager、TimelineController 等）。这使得 测试更容易并防止意外耦合。(#4769, #4613, #3331, #1190)

4. Track 与 Track 事件

  • ❌ 除 async slice 之外的任何东西使用 ChromeSliceTrack 基类 → ✅ 选择正确的 track 基类（ProcessCounterTrack 用于 counter、 ThreadStateTrack 用于调度状态、AsyncSliceTrack 用于 async slice）。 每个基类提供正确的渲染和交互语义。*(#5595, #5436, #5243)*

  • ❌ Track 类型注册缺失或类型名称不一致 → ✅ 注册每个新 track 类型且 名称一致。Track 类型名称必须跨 TrackDescriptor.type 和 UI 注册 保持一致；使用 registerTrack。 (#5436, #5595, #5243, #4278, #5188)

5. SQL 与查询

  • 🔁 ❌ 直接写裸 SQL 字符串 → ✅ 尽可能使用 stdlib 表/视图/函数；必要 时使用 perfetto-query-lang DSL。DSL 提供参数化、类型安全和构建时 检查。*(#4769, #5436, #5243, #5182, #4487, #3237)*

  • ❌ 查询中使用 SELECT * 的 MATERIALIZED 视图 → ✅ 仅物化所需列。 MATERIALIZED 视图会在 trace 加载时物化所有列，即使 UI 只需要其中一列 也是如此，这会浪费内存。*(#5182, #4487)*

  • ❌ 连接复杂查询时在 JS 代码中手动拼接 SQL 和参数 → ✅ 在 DSL 中完全 组合查询。避免 SQL 注入和参数错配。*(#4487)*

6. CSS 与样式

  • ❌ 硬编码颜色 → ✅ 使用主题变量（--pf-*、--pf-colour-*， theme.ts 中的 Material Design 色调板）。仅在绝对必要时指定背景颜色， 以保持 UI 可主题化。*(#2615, #5153, #4991, #4185, #1097)*

  • ❌ 通过字符串拼接/模板字面量构建 classNames → ✅ 使用 classNames() 工具函数。(#3693, #2615, #1406)

  • ❌ 冗余声明（font-family/size、cursor、对 pre/code/button 的 全局元素重置）→ ✅ 让它们继承 / 删除重复。不要对页面元素设置 font-family——让它从根继承以便主题生效。 (#5222, #3121, #2826, #2394)

  • ❌ 一次性 CSS 变量 / 将功能样式堆积到共享全局 SCSS 文件中 → ✅ 内联 值；以聚合 import 启动一个专属根 SCSS。(#1097, #5243)

  • 杂项布局技巧： align-items: baseline 对齐 widget 文本基线； align-self: center 对齐偏移的图标（不要更改容器）；box-shadow 用于 focus 轮廓（避免加粗边框引起的布局偏移）；将 <input> 嵌套在其 <label> 内以获得原生无障碍/点击行为；固定高度可滚动菜单以避免异步 加载时的闪烁；避免使用 <strong> 和 translate/transform 悬停技巧来 强调。*(#5222, #3121, #3946, #3844, #3817, #2615, #2093)*

7. TypeScript 与类型

  • 🔁 ❌ null → ✅ undefined。代码库在此处严格一致；使用 ?: 可选 语法。唯一允许的 null 是查询结果中的 SQL NULL——即便如此，在 UI 代码中也优先映射为 undefined。*(#4192, #3599, #3188, #1389)*

  • 🔁 ❌ 类型断言（as）和非空断言 ! → ✅ 让 TS 窄化类型，或大声断言 不变量。TS 在 .filter、Object.entries 等之后会窄化；不要断言 本身是 unknown 的类型。将对无类型数据的大量断言替换为 Zod 解析。 使用 assertExists(x) 而非 x!，使开发者错误的 null 立即失败。使用 LONG/LONG_NULL + Time.fromRaw() 而非强制转换 SQL 值。在可用时 传递具体实现类型（TraceImpl）以避免下游强制转换。不要将查找包装在 Object(...) 中——它会将类型抹除为 any；改为给查找表添加类型。 (#4767, #4890, #4769, #4653, #4613, #3331, #2580)

  • 🔁 ❌ 扁平接口带许多始终可选的字段，其有效组合未被强制执行 → ✅ 基于 kind/status/type 标签的可辨识联合类型（与共享字段交叉）。让 类型系统强制每个变体需要哪些字段。*(#4192, #3331)*

  • 🔁 **❌ 可变数组参数 → ✅ readonly T[]**（包括 getter/setter 中）用于 不修改的集合。将接口 attrs 标记为 readonly。 (#3188, #3058, #2039)

  • ❌ 穷举 switch 的 default 中缺少 assertUnreachable() → ✅ 添加它， 使编译器在添加新的联合类型 case 时报错。*(#2615, #1565)*

  • ❌ 用新的包装接口重复已有类型，或将数据路由通过宽泛/通用接口再重新 解析 → ✅ 直接返回已有/正确的特定类型（例如返回 QueryResult； 返回 AndroidInputEventSource 而非解析回来的通用 args）。在接受普通 number 的 API 上去掉品牌类型（SliceSqlId）。 (#4844, #4767, #4729)

  • ❌ == null 比较 / 通用 == / 过早合并到 sentinel 值 → ✅ if (!x)、x ?? 0、nullish 运算符。注意 JS falsy 陷阱：0 和 null/undefined 都是 falsy——通过 (number|null)[] 携带 null， 将真实的 0 显式显示，而非将其渲染为空白。 (#1582, #1389, #1376)

  • 较小类型规范：没有 await 的函数不要标记 async；添加 declare global { interface Window {...} } 扩充以注入到 window （不要强制转换）；在类型繁重的代码中为中间变量添加显式注解；不要强制 接口属性为 getter（使用普通 readonly）；不要冗余初始化构造函数始终 赋值的字段；当其他分支返回值时，优先使用显式 return undefined 而非 裸 return；避免将数字键通过字符串来回转换；不要无意中扩大成员的 可见性（private→public）；不要添加始终被提供的可选参数。Zod schema 必须镜像实际的嵌套数据形状。 (#4278, #4192, #2644, #2615, #1763, #2897, #5222, #3058)

8. 命名与约定

  • 🔁 ID 遵循既定约定。设置/注册 ID：反向 DNS，核心属 dev.perfetto.*， 外部 plugin 使用贡献者组织的命名空间（com.meta.*、com.android.*）。 命令 ID：省略冗余的 plugin 名称，使用稳定的命名空间化动词形式，以便 命令可在 plugin 之间移动。不要将设置的 ID 与 plugin 名称绑定—— 与 plugin 无关的 ID 使其可在不丢失用户保存值的情况下重定位。不要对仅在 你自己的 plugin 内注册的内容过度命名空间化。 (#5787, #5761, #3362, #3259, #2036, #4653, #4767, #5436)

  • 🔁 变量/成员使用 camelCase；文件名使用 snake_case；不使用匈牙利 命名法。不使用 snake_case 变量（trace_processor_id），不使用 缩写前缀（uMachineIds → uniqueMachineIds），私有方法不使用前导 下划线，仅在参数有意未使用时添加下划线前缀。 (#2615, #1155, #1913, #1389, #1565, #4737)

  • 以事物本身/所做的事情命名，在合适的抽象层级。只显示（从不切换）的 函数命名为 showX，而非 toggleX。一次性 init 函数命名为 load/initialize，而非 build（读起来像每帧 Mithril 辅助函数）。 Widget 使用标准术语命名（treemap、scatterplot），而非 treemap_chart。当 widget 范围扩大时重命名它。当上下文中所有内容都 已有限定词时，删除冗余限定词。不在一个标识符中使用两个同义词 （canvas/container）。布尔值以其守卫的内容命名 （isProcessingQueue，而非 isProcessing）。如果预期复用，按通用 概念命名字段。使命令名称自解释/可发现。 (#3169, #4192, #4761, #4729, #4278, #1406, #1302, #4380, #5566)

  • 机械约定：每个新文件包含版权头（新文件使用当前年份；编辑时不要 改动年份）；仓库内模块使用相对 import 标识符；使用 // 行注释在行上方 或右侧，不使用 /* */；运行 ui/format-sources（不要手动格式化）； 检查新默认快捷键是否与现有全局快捷键冲突。 (#2394, #4582, #4769)

9. 错误处理

  • 🔁 ❌ 吞掉错误（空 catch、console.warn、返回 undefined）→ ✅ 让失败可见。控制台消息在实践中会被忽略。对于开发者错误，让它崩溃 （保留 stacktrace + 分析）；对于预期失败，向用户显示消息（例如 showModal）。空 catch 会在重构期间静默吞噬拼写错误。 (#4582, #4344, #4192)

  • ❌ 在异常引发处捕获 → ✅ 冒泡到合适的粒度。让调用方的循环捕获并跳过 失败项，而非在辅助函数内部逐项吞噬。考虑返回 Result<T>（StatusOr 风格） 以强制调用方显式处理失败。*(#4192)*

  • ❌ 宽泛的 try 块 → ✅ 仅包裹实际可能抛出的调用，而非周围的设置 代码。*(#4192)*

  • 🔁 ❌ 非空 ! / 静默回退到全局默认值 → ✅ 对开发者错误不变量使用 assertExists()。例如 assertExists(canvasElement)——绝不意外回退到 document；对不可能状态使用 assertExists(queue.shift()) 而非防御性 的 if (!blob) continue。*(#4994, #4761, #2580, #1302)*

  • ❌ 无超时的网络 fetches / 阻塞核心流程 → ✅ 使用 fetchWithTimeout / orTimeout 并显示软错误。不要依赖浏览器的约 5 分钟默认超时；核心 路径上的细粒度延迟 fetches 有"UI 在流程中悄悄停止工作"的风险。（也不 要过于激进地设置超时——10 秒以上，人们会使用热点网络。） (#4192)

  • ❌ try/catch 没有明确处理的目标，或对预期的空条件发出警告 → ✅ 保持明确。不要对首次运行的 localStorage 状态缺失发出警告。在确认 失败路径已被处理之前，不要移除可能失败查询周围的已有 try/catch 守卫 （这可能导致非 Google 环境出错）。*(#2615, #4115, #1513)*

  • ❌ 一行有问题就中止整个批次 → ✅ 跳过/丢弃该行并继续。确保处理器在 处理中途抛出异常时不会使共享状态不一致。*(#2897, #1302)*

10. 性能

  • 虚拟化：对不需要可变行高且高度受限的 grid 启用虚拟化；保留 grid 自己的滚动包装器（pf-sql-table）使其可以虚拟化——不要让整个详情 shell 滚动。如果对查询进行虚拟滚动很慢，先物化一个支持表（datagrid 本身不 物化任何内容）。*(#3455, #3423, #5276)*

11. PR 规范与范围

  • 🔁 ❌ 将不相关的重构、附带重命名、多余字段、空白调整捆绑到一个功能 PR 中 → ✅ 保持 diff 最小化并限定在其目的范围内。不相关的变更增加 噪音，使回归难以发现。瞄准 MVP；将重构和大型 mock 拆分到单独的 PR 中。 不要重命名未做其他更改的文件中的变量/标识符。 (#5222, #4641, #3736, #2666, #5649, #3170)

  • 🔁 ❌ 死代码/推测性代码 → ✅ YAGNI。删除未使用的函数、参数、循环 计数器和"以防万一"的代码行（通过删除并确认行为不变来验证）。删除 遗留的 console.log()——删除代码而非记录 "not implemented"。 (#5222, #5436, #1389, #1097)

  • 添加新 plugin/组件时，附带一个包含联系方式的 OWNERS 文件。 (#5284)
